**Scope:** Client-side connection methodology, NAT punchthrough, hairpin detection, and Linux stability

## 1. Connection Methodology Summary

The LMP Client uses three primary paths to reach a server:

| Path | Trigger | Files |
|------|---------|-------|
| **Direct Connect** | Favorite / Custom / CLI (`-lmp`) | `NetworkConnection.cs` |
| **LAN Bypass** | Public list — same network detected | `NetworkServerList.cs` |
| **NAT Punchthrough** | Public list — WAN server | `NetworkServerList.cs` + `Lidgren/NetNatIntroduction.cs` |

**Flow:**
1. `ConnectToServer(hostname, port, password)` resolves DNS, detects hairpin NAT, then delegates to a background task.
2. The background task starts `NetClient`, calls `client.Connect(endpoint)`, and busy-waits (50 ms loop) for handshake completion.
3. If Lidgren handshake fails, `DiagnoseTimeout` runs an ICMP ping to distinguish "host down" from "UDP blocked".
4. For public servers, `IntroduceToServer` decides LAN vs NAT. LAN connects directly to internal endpoints. WAN sends a `MsIntroductionMsgData` to master servers, which relay Lidgren NAT introduction messages.
5. On `NatIntroductionSuccess`, the client calls `ConnectToServer(punchedEndpoint)`.

## 2. Critical Bugs

### Bug 2.1 — IPv6 LAN Detection is Always False
**Location:** `NetworkServerList.cs:191`  
**Issue:** `if (ownBytes == serverBytes)` compares array **references**, not contents. `Array.Resize` creates new arrays, so this is always `false`.  
**Impact:** IPv6 LAN servers never bypass NAT; they always attempt punchthrough and often fail.

### Bug 2.2 — Hairpin Detection Blocks on Synchronous, Uncached HTTP
**Location:** `NetworkConnection.cs:132` → `LunaNetUtils.GetOwnExternalIpAddress()`  
**Issue:** On every direct connect to a public IP, the code calls `ip.42.pl`, `api.ipify.org`, `httpbin.org`, and `checkip.dyndns.org` **synchronously** with no timeout.  
**Impact:** Connection thread freezes for 30–100 seconds on offline or restrictive networks. International users suffer the most.

### Bug 2.3 — Unsafe Local Port Fallback
**Location:** `NetworkConnection.cs:309` (`FindLocalServerPort`)  
**Issue:** If the exact external port and `8800` are not found, it picks the numerically closest non-system UDP port.  
**Impact:** Can select Discord, Steam, or another application's port, causing a connection to the wrong service or a confusing failure.

### Bug 2.4 — No Fallback When NAT Punch Fails
**Location:** `NetworkServerList.cs:152`  
**Issue:** After the 10-second NAT timeout, the code only logs and updates UI status. It does **not** fall back to direct connect on the external endpoint.  
**Impact:** Users behind symmetric NAT or with master-server issues have no recovery path.

### Bug 2.5 — Missing Endpoint Validation on NAT Success
**Location:** `NetworkServerList.cs:204` (`HandleNatIntroductionSuccess`)  
**Issue:** The token is validated, but `msg.SenderEndPoint` is never checked against the expected `ServerInfo.ExternalEndpoint`.  
**Impact:** A rogue or misconfigured master server could redirect the client to an arbitrary host.

### Bug 2.6 — Sequential DNS with No Happy Eyeballs
**Location:** `NetworkConnection.cs:187`  
**Issue:** If DNS returns IPv6 first and the path is dead, the client waits for the full Lidgren timeout before trying IPv4.  
**Impact:** Unnecessary connection delays on dual-stack networks with broken IPv6 routing.

## 3. Linux-Specific Freeze Root Causes

### Freeze L1 — Unity API Called from Background Threads
**Location:** `NetworkConnection.cs:50-57` (`Disconnect`)  
**Issue:** `Disconnect()` reads `HighLogic.LoadedSceneIsEditor` and `HighLogic.LoadedSceneIsFlight` from the receive thread, send thread, and connection thread. These are Unity APIs and **must** run on the main thread. On Linux/Mono this reliably deadlocks or crashes the render thread.

### Freeze L2 — Synchronous Network I/O on the UI Thread
**Location:** `NetworkConnection.cs:77` (`ConnectToServer` hostname overload)  
**Issue:** The **calling thread** (Unity main thread) performs `Dns.GetHostAddresses()`, `GetOwnExternalIpAddress()`, and `FindLocalServerPort()` **before** starting the background task.  
**Impact:** KSP UI freezes before the connection background task is even started.

### Freeze L3 — `Ping` Hangs on Linux/Mono
**Location:** `NetworkConnection.cs:387-390` (`DiagnoseTimeout`)  
**Issue:** `new Ping()` requires `CAP_NET_RAW`. Without it, Mono's implementation often **blocks indefinitely** instead of throwing `PlatformNotSupportedException`. The 3000 ms timeout only applies after socket creation; creation itself hangs.  
**Impact:** The connection background task deadlocks, leaving the UI stuck on the connecting screen forever.

### Freeze L4 — `ResetNetworkSystem` Called from Any Thread
**Location:** `MainSystem.cs:47-56` and `NetworkMain.cs:101`  
**Issue:** `NetworkState = ClientState.Disconnected` triggers `ResetNetworkSystem()`, which calls `ClientConnection.Shutdown()` and waits for threads. If a background thread triggers this while another is stuck in socket I/O, it creates a cross-thread deadlock.

### Freeze L5 — `NetworkInterface.GetAllNetworkInterfaces()` Hangs
**Location:** `LunaNetUtils.cs:193` (`GetNetworkInterface`)  
**Issue:** On Linux with systemd-networkd, Docker bridges, or WireGuard, enumerating interfaces can take 5–10 seconds or hang entirely if a tunnel is in a bad state. This is called synchronously during connection setup.

### Freeze L6 — `WebClient` Default Timeout is 100 Seconds
**Location:** `LunaNetUtils.cs:165` (`TryGetIpAddress`)  
**Issue:** `WebClient` has no timeout configured. On restrictive networks, the external IP lookup hangs the caller for over a minute.

## 4. Robustness Issues (International / High-Latency)

| Issue | Location | Impact |
|-------|----------|--------|
| Aggressive 15 s Lidgren timeout | `SettingsStructures.cs:30` | Too short for trans-oceanic/satellite links (300–600 ms RTT + loss). |
| Misleading ICMP diagnostics | `NetworkConnection.cs:372` | Many cloud hosts block ICMP. "Host unreachable" is logged even when the server is alive and only UDP is firewalled. |
| No per-endpoint adaptive timeout | `NetworkConnection.cs:166` | Lidgren retries internally, but the app layer gives up after one pass. |
| Master server single point of failure | `NetworkServerList.cs:108` | Public servers cannot be reached without a working master server. |
| Blast to all master servers | `NetworkSender.cs:89` | Duplicate NAT traffic and race conditions. |

## 5. Proposed Fix Plan

### Phase 1 — Critical Correctness
1. **Fix IPv6 LAN detection** — replace `ownBytes == serverBytes` with `SequenceEqual` or bitwise comparison.
2. **Remove unsafe port fallback** — `FindLocalServerPort` should return `-1` if no LMP listener is found on the expected port or default `8800`. Hairpin redirect only proceeds with a confirmed match.
3. **Add NAT punch fallback** — when the 10-second NAT timer fires, automatically attempt direct connect to the server's external endpoint before giving up.
4. **Validate NAT success endpoint** — store the expected `ServerInfo.ExternalEndpoint` during introduction and verify `msg.SenderEndPoint` matches before connecting.
5. **Log full connection decision tree** — log every branch: LAN check result, method selected, endpoint attempted, and result.

### Phase 2 — Performance & Latency
6. **Cache external IP + add WebClient timeout (3 s)** — cache the result for the session. Log cache hits/misses. If all lookups fail, assume "not hairpin" and continue.
7. **Move all connection setup off the UI thread** — DNS, external IP lookup, UDP listener scanning, and hairpin detection must run inside the background task. Only update `MainSystem.Singleton.Status` via thread-safe means.
8. **Implement Happy Eyeballs-style parallel connect** — for DNS results with multiple addresses, start IPv4 and IPv6 connection attempts with a 250 ms stagger, and take the first successful handshake.
9. **Raise default timeout to 25 s** — align with Lidgren's own default for high-latency paths.
10. **Smart master server selection** — pick one master by round-robin or lowest historical latency, and failover to others instead of blasting all simultaneously.

### Phase 3 — Linux Stability
11. **Thread-safe `Disconnect`** — remove `HighLogic` access from `Disconnect()`. Set a flag (`QueueDisconnectAction`) and let `MainSystem.Update` handle `ForceQuit` / `DisplayDisconnectMessage` on the main thread.
12. **Linux-safe Ping** — replace ICMP with a lightweight UDP probe, or wrap `Ping` in `Task.Run` with a hard 4-second timeout and catch everything. If it expires, log `"Ping probe timed out (platform may block ICMP)"` and continue.
13. **Main-thread-only network teardown** — ensure `NetworkMain.ResetNetworkSystem()` is only invoked from `MainSystem.Update`. Background threads should set `NetworkState = DisconnectRequested`.
14. **Timeout-wrap interface enumeration** — wrap `GetActiveUdpListeners()`, `GetAllNetworkInterfaces()`, and `GetIPProperties()` in try-catch with a 2-second watchdog. On timeout, abort the scan and fall back to conservative defaults.
15. **Add cancellation token to connection task** — pass a `CancellationToken` through the connection flow and check it in busy-wait loops. If the user clicks Cancel or a 30-second overall timeout fires, the token cancels and the task exits cleanly.
16. **Safe socket shutdown on Linux** — ensure the receive thread has exited (or is aborted as a last resort) before calling `ClientConnection.Shutdown` from the main thread, preventing `recvfrom` deadlocks.
